# AGENT.md（Codex向けガイド）

このリポジトリは、修士研究（Text2KG）の実験・解析・評価を再現可能に回すためのコードベースです。  
あなた（コーディングエージェント）は、**研究の再現性・ログ・入出力の明確さ**を最優先に、壊れにくい実装をしてください。
英語で考え実装した後、説明は全て日本語で行ってください。

---

## 0. 最重要ルール（絶対）

- **CLI引数（argparse等）で定義する全ての引数 argument には必ず `default` を付けること。**
  - `required=True` の運用は原則禁止（例外は「本当に無いと実験が成立しない識別子」など）。その場合でも極力 `default` + 検証エラーで案内する。
- 既存仕様・ディレクトリ構成・入出力形式（JSON/JSONL/CSV）を勝手に変えない。
- 研究コードなので、実行結果が変わりうる要因（乱数・並列・依存バージョン）を放置しない。
- 長時間処理は必ず進捗表示を付ける（例: tqdm）。学習やトレーニングは進捗バー必須。
- 可能な限り高速に（無駄な全件読み込み、無駄な正規表現、無駄なループを避ける）。
- **複数人が共用で使うサーバーなので、`apt upgrade`など環境に影響するコマンドは絶対に実行してはならない。**

---

## 1. 目的（このリポジトリがやりたいこと）

- Text2KGBench 和訳データを対象に、テキストから Knowledge Graph を抽出・評価する。
- CKY表（チャート）上で候補を列挙し、BERT系モジュールで
  - 述語-項関係
  - 連体修飾
  - Modification
  を**優先度付き**で判定する。
- オントロジー（domain/range）に整合するかを検証し、出力（triples / candidates / logs）を生成する。

---

## 2. 作業の基本方針（エージェントの振る舞い）

### 2.1 変更の進め方
- 変更は **最小単位**で行う（「1機能=1コミット」相当の粒度を意識）。
- 既存ファイルがある場合は、構造・命名を尊重し、局所的に改善する。
- 破壊的変更（I/O形式変更・キー名変更・ディレクトリ移動）は禁止。必要なら互換レイヤを追加。

### 2.2 再現性
- 乱数を使うなら seed を一箇所に集約して固定できるようにする。
- 実験出力は「実験バージョン」単位でディレクトリ分離し、ログ・設定も保存する。
  - 例: `out/exp_<yyyymmdd>_<tag>/` のような形（リポジトリ既存の流儀があればそれに従う）

### 2.3 ログ
- 標準出力だけに頼らず、`log.jsonl` や `run.log` を保存する。
- エラー時は「どの入力」「どのレコード」「どのステップ」で落ちたかが追える情報を出す。

## 3. 実装規約（Python）

### 3.1 CLI（argparse）
- **全引数に default を付けること（最重要ルールの再掲）。**
- デフォルトは「このリポジトリで現実的に動く値」を置く（不明なら安全な相対パス・一般的な値）。
- `--in_path`, `--out_dir` のように I/O が明確な命名にする。
- `--dry_run` を付けられる設計は歓迎（必須ではない）。

### 3.2 I/O
- JSONLは原則ストリーミング処理（行ごとに読む）。巨大データでメモリ死しないようにする。
- 出力ファイルは上書き衝突しない設計にする（サフィックス、実験IDなど）。
- データのパス指定は、特に指示がない限り **ファイル名/相対パス**を使う。

### 3.3 速度
- ループの内側で重い初期化（モデルロード、辞書ロード、正規表現compile）をしない。
- 同じ計算を繰り返さない（キャッシュ、前処理、ベクトル化）。
- 可能なら `multiprocessing` / `concurrent.futures` を使うが、再現性とログを壊さない範囲で。

### 3.4 進捗表示
- 長いforループには tqdm。
- 学習・推論バッチ処理には tqdm（エポック、ステップ、ファイル単位のどれか）。

### 3.5 例外処理
- 入力不正は「どの行/どのID/どのファイル」が原因かを含めて例外化する。
- ただ握りつぶさない。スキップするならスキップレポートを残す。

## 4. プロジェクト固有の前提（重要）

### 4.1 CKY表・候補列挙
- CKYセル内の判定は優先度付き：
  1) 述語-項関係  
  2) 連体修飾  
  3) Modification  
- 既存コードにこの順序があるなら壊さない。改善は「根拠」と「影響範囲」を明確に。

### 4.2 オントロジー整合
- domain/range による整合性チェックを前提とする。
- 文字列一致だけで決めず、必要ならID（QID/PID）や正規化を利用する。

## 6. 生成物の品質基準（レビュー観点）

- 実行一発で再現できる（依存・引数・入出力が明確）。
- ログが残り、失敗原因が追える。
- 大規模データでメモリ爆発しない。
- 余計な仕様変更がない。
- 速度面で致命的なムダがない。

---

## 7. ありがちな地雷（踏むな）

- default無しの引数を追加する（絶対ダメ）。
- JSONLを全部 `list()` してメモリ死。
- ループ内で毎回モデルロード。
- 出力を上書きして実験結果を破壊。
- 例外を握りつぶして「静かに壊れる」挙動にする。
